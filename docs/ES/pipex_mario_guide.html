PIPEX ADVENTURE

¡Tu misión de 9 días para dominar los pipes! 🎮

🎯 ¿Qué es Pipex?

Pipex es tu desafío para recrear el comportamiento de los pipes (|) de UNIX. Vas a implementar cómo funciona internamente el símbolo | cuando encadenas comandos en la terminal.

📋 Concepto Básico

En la terminal cuando ejecutas:

< archivo.txt grep "hola" | wc -l > salida.txt

Tu programa debe hacer exactamente lo mismo:

./pipex archivo.txt "grep hola" "wc -l" salida.txt

🔧 Funciones Principales que Usarás

Función Para qué sirve
pipe() Crear la tubería de comunicación entre procesos
fork() Crear procesos hijos (duplicar el proceso actual)
execve() Ejecutar los comandos del sistema
dup2() Redireccionar entrada/salida (stdin/stdout)
open() Abrir archivos para lectura/escritura
close() Cerrar file descriptors
wait()/waitpid() Esperar a que terminen los procesos hijos
access() Verificar si un archivo existe y tiene permisos

✅ ❌ Permitido vs Prohibido

Funciones Permitidas

  • open, close, read, write
  • malloc, free
  • perror, strerror
  • access
  • dup, dup2
  • execve
  • exit
  • fork
  • pipe
  • unlink
  • wait, waitpid
  • ft_printf o cualquier función que TÚ hayas codificado
  • Usar tu libft (si la tienes)

🚫 Prohibido

  • system() - Debes usar execve()
  • popen() - Ya hace el trabajo por ti
  • Funciones de la biblioteca estándar que no estén en la lista permitida
  • Memory leaks - Todo malloc debe tener su free
  • Crashes - Segfaults, bus errors, double free, etc.
  • Errores de la Norm - Tu código debe cumplir la Norminette
Importante: Si no estás seguro de cómo manejar un error, hazlo de la misma forma que lo hace el shell con el comando equivalente.

✍️ Qué Implementar

main.c:

  • Validar número de argumentos (debe ser 5: ./pipex file1 cmd1 cmd2 file2)
  • Llamar a la función principal de pipex
  • Retornar código de salida apropiado

pipex.c:

  • create_pipe() - Crea el pipe y maneja errores
  • pipex() - Función principal que coordina todo

child.c:

  • first_child() - Proceso para ejecutar cmd1
    • Abre file1 en modo lectura
    • Redirige stdin desde file1 con dup2()
    • Redirige stdout al pipe con dup2()
    • Cierra file descriptors no necesarios
    • Ejecuta cmd1 con execve()
Ejemplo de flujo first_child():

1. fork() crea el hijo
2. Dentro del hijo:
   - fd = open("file1", O_RDONLY)
   - dup2(fd, STDIN_FILENO)      // stdin ahora lee de file1
   - dup2(pipe_fd[1], STDOUT_FILENO) // stdout ahora escribe al pipe
   - close(fd), close(pipe_fd[0]), close(pipe_fd[1])
   - execve(path_cmd1, args_cmd1, envp)
Consejo del Día: Cierra TODOS los file descriptors que no necesites. Un fd abierto puede causar que el programa se quede colgado esperando.

📁 Actualizar Archivos

│   ├── pipex.c       (completar lógica)
│   └── (resto de archivos)

✍️ Qué Implementar

child.c - Añadir:

  • second_child() - Proceso para ejecutar cmd2
    • Abre file2 en modo escritura (crea si no existe)
    • Redirige stdin desde pipe con dup2()
    • Redirige stdout a file2 con dup2()
    • Cierra file descriptors no necesarios
    • Ejecuta cmd2 con execve()

pipex.c - Completar:

Flujo completo en pipex():

1. pipe(pipe_fd)
2. pid1 = fork()
   - Si es hijo1: first_child()
3. pid2 = fork()
   - Si es hijo2: second_child()
4. En el padre:
   - close(pipe_fd[0])
   - close(pipe_fd[1])
   - waitpid(pid1, &status, 0)
   - waitpid(pid2, &status, 0)
Importante: El padre DEBE cerrar ambos extremos del pipe antes de hacer wait. De lo contrario, el segundo hijo puede quedar esperando indefinidamente.

✍️ Qué Implementar

Diferencias del Bonus:

  • Ahora argc es variable: ./pipex file1 cmd1 cmd2 cmd3 ... cmdN file2
  • Necesitas crear N-1 pipes (donde N = número de comandos)
  • Necesitas N procesos hijos
  • Cada hijo se conecta a 2 pipes (entrada y salida), excepto el primero y el último

Estrategia:

Para 3 comandos (cmd1 | cmd2 | cmd3):

file1 → CHILD1 → pipe1 → CHILD2 → pipe2 → CHILD3 → file2

Necesitas:
- 2 pipes: pipe1[2], pipe2[2]
- 3 forks
- Cada hijo cierra los pipes que no usa
  • Crea un array de pipes: int pipes[n_cmds - 1][2]
  • Usa un bucle para crear todos los pipes
  • Usa otro bucle para crear todos los procesos hijos
  • Cada hijo calcula qué pipes usar según su índice

✍️ Here_doc - ¿Qué es?

En shell, here_doc permite dar input directamente sin archivo:

cat << EOF | wc -l
linea 1
linea 2
linea 3
EOF

Tu programa debe soportar:

./pipex here_doc LIMITER cmd1 cmd2 file

Equivale a:
cmd1 << LIMITER | cmd2 >> file

# Nota: >> es append, no sobrescribe

Implementación:

  • Detecta si argv[1] == "here_doc"
  • Lee líneas de stdin hasta encontrar LIMITER
  • Usa get_next_line() si la tienes, o lee con read()
  • Escribe todo en el pipe para que cmd1 lo lea
  • Abre file2 con flag O_APPEND en lugar de O_TRUNC

⭐ Explicación del Bonus

1️⃣ Múltiples Pipes

Soportar cualquier número de comandos encadenados:

./pipex file1 cmd1 cmd2 cmd3 cmd4 file2

Equivale a:
< file1 cmd1 | cmd2 | cmd3 | cmd4 > file2

Concepto Visual:

file1 → [CMD1] → pipe1 → [CMD2] → pipe2 → [CMD3] → pipe3 → [CMD4] → file2

- N comandos necesitan N-1 pipes
- Cada comando (excepto primero y último) lee de un pipe y escribe a otro
- Primer comando: lee de file1, escribe a pipe
- Último comando: lee de pipe, escribe a file2
- Comandos del medio: leen de pipe[i-1], escriben a pipe[i]

2️⃣ Here_doc

Leer input directamente del terminal en lugar de un archivo:

./pipex here_doc EOF "cat" "wc -l" outfile

Ejecución:
$ ./pipex here_doc EOF "cat" "wc -l" outfile
> linea 1
> linea 2
> linea 3
> EOF

Resultado en outfile:
3

Diferencias con modo normal:

  • No abres file1, lees de stdin hasta encontrar el LIMITER
  • Abres file2 con O_APPEND en vez de O_TRUNC (>> en vez de >)
  • El primer comando lee de un pipe donde tú escribiste el input
